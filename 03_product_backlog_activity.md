# Product Backlog Activity

## Objective

The objective of this activity is to help students translate the product vision into a structured product backlog composed of user stories and acceptance criteria.

In the educational experience, the product backlog was used as an intermediate artifact between the product vision and later design and sustainability-oriented requirements activities. It helped student teams move from a broad understanding of the problem to a more organized set of development-oriented items.

## Learning goals

After completing this activity, students are expected to be able to:

- derive backlog items from the product vision;
- write user stories from the perspective of users or actors;
- express user needs rather than only technical solutions;
- clarify the value or benefit of each user story;
- organize backlog items according to priority or relevance;
- identify missing, redundant, overlapping, broad, or vague items;
- define acceptance criteria that support verification;
- connect backlog items to later UML modeling and sustainability-oriented requirements activities.

## Student task

Each team should create a product backlog aligned with the product vision.

The backlog should include:

1. user stories;
2. priority or another form of organization among items;
3. acceptance criteria for each user story;
4. notes or comments when necessary.

The backlog should represent the most important functionalities of the proposed system and should remain coherent with the problem, users, scope, and expected value described in the product vision.

## Suggested template

| ID | User story | Priority | Acceptance criteria | Notes |
|---|---|---|---|---|
| US01 | As a [type of user], I want to [goal], so that [benefit or value]. | High/Medium/Low | Conditions that help verify whether the user story was satisfied. | Optional notes. |

## User story format

Students may use the following format:

> As a [type of user],
> I want to [goal],
> so that [benefit or value].

A user story should make clear:

- who the user or actor is;
- what the user wants to do;
- why this is important;
- what value or benefit is expected.

## Acceptance criteria

Acceptance criteria should help verify whether a user story has been satisfactorily addressed.

They may be written using the following format:

> Given [context],  
> When [action],  
> Then [expected result].

They may also be written as a checklist of verifiable conditions.

Acceptance criteria should avoid vague statements. They should make it possible to discuss, review, test, or inspect whether the expected behavior was achieved.

## Backlog overview review

After creating the backlog, each team should review it using the following questions:

- Is the backlog coherent with the product vision?
- Does the backlog include the most important functionalities of the system?
- Are there redundant, overlapping, or very similar items?
- Do important functionalities seem to be missing?
- Are the items organized by priority or by another meaningful criterion?

## User story quality review

Each team should review all user stories using the following questions:

- Are the user stories written from the perspective of a user or actor?
- Do the user stories follow an adequate format?
- Do the user stories make clear what the user wants to do?
- Do the user stories make clear why this is important to the user?
- Are there user stories that are too broad, generic, or vague?
- Are there user stories that are too technical, focused on the solution rather than on the user's need?
- Do the user stories seem small enough to be understood and implemented?
- Does the backlog present priorities or some form of organization among items?

## Acceptance criteria review

Each team should review the acceptance criteria using the following questions:

- Do the user stories have acceptance criteria?
- Do the acceptance criteria make it possible to verify whether the user story was satisfied?
- Are the acceptance criteria clear enough to support review or testing?
- Are there acceptance criteria that are too vague, subjective, or difficult to assess?

## Common issues to avoid

Students should pay special attention to the following issues:

- backlog items that are not aligned with the product vision;
- user stories written as technical tasks instead of user needs;
- user stories without an explicit user, goal, or benefit;
- user stories that are too broad to be discussed or implemented;
- duplicated or overlapping items;
- missing priorities;
- missing or vague acceptance criteria;
- acceptance criteria that do not support verification.

## Connection to later activities

The product backlog is used as input for later activities in the course.

In the UML modeling activity, students use backlog items to identify relevant workflows, system states, responsibilities, and components.

In the sustainability-oriented requirements activity, students revisit the backlog to identify implicit quality concerns. For example, a user story about notifications may raise concerns about alert fatigue; a user story about data registration may raise concerns about duplicated work or data reuse; and a user story about access to information may raise concerns about accessibility, privacy, or digital literacy.

By reviewing the backlog through a sustainability-oriented lens, students are encouraged to move beyond the question of what the system should do and consider how the system should behave, who may be affected, and how quality-related concerns can be made explicit.
