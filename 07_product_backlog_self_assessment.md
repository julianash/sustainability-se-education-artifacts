# Product Backlog Self-Assessment Instrument

## Purpose

This document provides an anonymized and adapted version of the self-assessment instrument used to support the review of the product backlog artifact.

The purpose of the instrument is to help student teams evaluate whether their product backlog is coherent with the product vision, sufficiently complete, organized, understandable, and useful as a basis for later Software Engineering activities, including UML modeling and sustainability-oriented requirements analysis.

The instrument focuses especially on the quality of user stories and acceptance criteria.

## Anonymization note

The original form included identifying fields such as e-mail, student name, and project name. These fields were removed or generalized for double-anonymous review.

This version is provided as a reusable educational artifact.

## Suggested response scale

For each closed question, students may answer:

- Yes
- No
- Partially

Students are encouraged to add comments whenever they answer "No" or "Partially".

## 1. Backlog overview

The product backlog should be coherent with the product vision and should include the most important functionalities of the proposed system.

| Is the backlog coherent with the product vision? |  |  |  |  |
| Does the backlog include the most important functionalities of the system? |  |  |  |  |
| Are there redundant, overlapping, or very similar items? |  |  |  |  |
| Do important functionalities seem to be missing? |  |  |  |  |

### Open reflection

- Describe the problems identified in the backlog overview.

## 2. Quality of user stories

Each user story should be written from the perspective of a user or actor and should express a user need, goal, or expected value.

| Question | Yes | No | Partially | Comments |
|---|---|---|---|---|
| Are the user stories written from the perspective of a user or actor? |  |  |  |  |
| Do the user stories follow an adequate format? |  |  |  |  |
| Do the user stories make clear what the user wants to do? |  |  |  |  |
| Do the user stories make clear why this is important to the user? |  |  |  |  |
| Are there user stories that are too broad, generic, or vague? |  |  |  |  |
| Are there user stories that are too technical, focused on the solution rather than on the user's need? |  |  |  |  |
| Do the user stories seem small enough to be understood and implemented? |  |  |  |  |
| Does the backlog present priorities or some form of organization among items? |  |  |  |  |

## Expected user story format

Students may use the following format:

> As a [type of user],  
> I want to [goal],  
> so that [benefit or value].

A user story should make clear:

- who the user or actor is;
- what the user wants to do;
- why this is important;
- what value or benefit is expected.

## 3. Acceptance criteria

Acceptance criteria should help verify whether a user story has been satisfactorily addressed.

| Question | Yes | No | Partially | Comments |
|---|---|---|---|---|
| Do the user stories have acceptance criteria? |  |  |  |  |
| Do the acceptance criteria make it possible to verify whether the user story was satisfied? |  |  |  |  |
| Are the acceptance criteria clear enough to support review or testing? |  |  |  |  |
| Are there acceptance criteria that are too vague, subjective, or difficult to assess? |  |  |  |  |

## Suggested acceptance criterion format

Acceptance criteria may be written using the following format:

> Given [context],  
> When [action],  
> Then [expected result].

They may also be written as a checklist of verifiable conditions.

## Common issues to avoid

Students should pay attention to the following issues:

- backlog items that are not aligned with the product vision;
- missing important functionalities;
- duplicated or overlapping items;
- user stories written as technical tasks;
- user stories without an explicit user, goal, or benefit;
- user stories that are too broad, generic, or vague;
- user stories that are too large to be understood or implemented;
- lack of priorities or organization among items;
- missing acceptance criteria;
- acceptance criteria that are not verifiable;
- acceptance criteria that are subjective or ambiguous.

## Connection to later activities

The product backlog is used as input for later activities in the course.

In the UML modeling activity, students use backlog items to identify relevant workflows, states, components, responsibilities, and interactions.

In the sustainability-oriented requirements activity, students revisit the backlog to identify implicit quality and sustainability concerns.

For example:

- a user story about notifications may raise concerns about alert fatigue;
- a user story about data registration may raise concerns about duplicated work or data reuse;
- a user story about access to information may raise concerns about accessibility, privacy, or digital literacy;
- a user story about reports may raise concerns about decision support, clarity, workload, or data quality.

By reviewing the backlog through a sustainability-oriented lens, students are encouraged to move beyond functional behavior and consider how the system should behave, who may be affected, and how quality-related concerns can be made explicit.
