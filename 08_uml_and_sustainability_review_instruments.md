# UML and Sustainability Review Instruments

## Purpose

This document provides complementary review instruments for UML models and sustainability-oriented requirements.

The product vision and product backlog self-assessment instruments are provided in separate files. This document focuses on two later moments of the educational experience:

1. review of UML models;
2. review of sustainability-oriented requirements.

The purpose is to support formative feedback and help student teams refine their artifacts before the final presentation and discussion.

## Suggested response scale

For each closed question, students may answer:

- Yes
- No
- Partially

Students are encouraged to add comments whenever they answer "No" or "Partially".

# Part 2: Sustainability-Oriented Requirements Review

## General review

The sustainability-oriented requirements should be connected to the project context, previous artifacts, and one or more sustainability dimensions.

| Question | Yes | No | Partially | Comments |
|---|---|---|---|---|
| Is the sustainability concern connected to the project context? |  |  |  |  |
| Is the sustainability dimension clearly indicated? |  |  |  |  |
| Is the requirement non-functional rather than merely functional? |  |  |  |  |
| Is the requirement specific enough to be discussed and reviewed? |  |  |  |  |
| Is the requirement aligned with the product vision, backlog, or UML models? |  |  |  |  |
| Is there at least one acceptance criterion? |  |  |  |  |
| Is the acceptance criterion assessable? |  |  |  |  |
| Is there at least one possible source of verification evidence? |  |  |  |  |
| Is the proposed evidence coherent with the requirement? |  |  |  |  |
| Could the requirement influence design, testing, implementation, or future maintenance decisions? |  |  |  |  |

## Review by sustainability dimension

Teams should consider whether they focused only on technical concerns or whether other dimensions were also considered.

| Dimension | Guiding review question | Notes |
|---|---|---|
| Technical | Did the team consider maintainability, modularity, interoperability, testability, reliability, or long-term evolution? |  |
| Social | Did the team consider accessibility, inclusion, equity, trust, collaboration, or social impact? |  |
| Economic | Did the team consider rework, costs, productivity, resource allocation, or long-term viability? |  |
| Environmental | Did the team consider data storage, resource use, redundant processing, paper reduction, device use, or computational efficiency? |  |
| Individual | Did the team consider autonomy, cognitive load, workload, privacy, well-being, attention, or digital fatigue? |  |

## Review of the sustainability-oriented requirements table

Each team should review the table produced in the activity.

| Element | Review question | Notes |
|---|---|---|
| Dimension | Is the selected dimension appropriate for the concern? |  |
| Sustainability concern | Is the concern specific to the project context? |  |
| Non-functional requirement | Does the requirement express a quality, constraint, or expected property of the system? |  |
| Acceptance criterion | Does the criterion help assess whether the requirement is satisfied? |  |
| Verification evidence | Is the proposed evidence plausible and coherent with the requirement? |  |

## Common issues to identify

Students should check whether their work presents any of the following issues:

- the concern is too generic;
- the requirement only restates a functionality;
- the requirement is disconnected from the project context;
- the sustainability dimension is unclear or inappropriate;
- the acceptance criterion is vague or subjective;
- the verification evidence does not actually help assess the requirement;
- the team focused only on technical sustainability;
- environmental sustainability was reduced only to paper reduction;
- individual or social impacts were ignored;
- the requirement is unrealistic for the course project;
- the activity was not connected to previous artifacts.

## Connection to formative feedback

These review instruments may be used by students, peers, or the instructor.

They can support:

- self-assessment;
- peer review;
- instructor feedback;
- classroom discussion;
- preparation for final presentations;
- revision of product vision, backlog, UML models, and sustainability-oriented requirements.

The goal is not only to check whether artifacts are complete, but also to help students understand Software Engineering as an iterative process of refinement, communication, analysis, and quality-oriented decision-making.
